# feat-gcm

## 目标
- 基于本地全部修改文件分析功能改动，而不是只看暂存区。
- 自动判断本次改动涉及多少个独立功能，并按功能拆分为多次提交。
- 每次提交只包含一个功能，提交信息必须简短，并且以 `feat:` 开头。
- 先给出完整提交计划让用户确认，再根据用户选择决定仅提交、提交并推送，或取消。

## 流程
1. 查看整个工作区的本地修改，包括：
   已跟踪文件的修改。
   已跟踪文件的新增或删除。
   未跟踪的新文件。
2. 不以当前暂存区为准，必须根据完整本地改动重新分析功能边界。
3. 基于 diff、文件分布和功能关联性，自动判断本次改动拆成几次提交最合理。
4. 拆分原则如下：
   单次提交只包含一个独立功能。
   不要把互不相关的功能混在同一次提交里。
   如果多个文件共同服务于同一个功能，应归为同一次提交。
   如果边界不够清晰，宁可少拆，也不要机械按文件数拆分。
5. 为每一次计划提交生成一句简短中文提交信息，格式固定为：
   `feat: <功能描述>`
6. 将全部候选提交信息按序号列出给用户确认，例如：
   `1. feat: 新增任务详情视频回放`
   `2. feat: 调整任务创建负责人选择`
7. 在列出提交信息后，统一给用户 3 个选项：
   `1` 表示按当前拆分方案依次提交。
   `2` 表示按当前拆分方案依次提交，并在全部提交完成后立即推送。
   `3` 表示取消。
8. 取消的含义必须明确执行为：
   不进行任何提交。
   不保留 skill 为拆分提交而新增的暂存区内容。
   最终暂存区应为空。
9. 如果用户选择 `1` 或 `2`，再开始按功能逐次暂存并提交：
   每次只暂存当前这一条功能对应的文件或 hunk。
   提交完成后再处理下一条功能。
   不允许一次性把所有修改都加到暂存区再连续提交。
10. 如果用户选择 `2`，则在全部提交成功后再执行推送，而不是每次提交后都推送。

## 约束
- 提交信息必须简短，一次提交只用一句话表达一个功能。
- 所有中文、英文、数字之间保持空格规范，标点符号中英文不混用。
- 如果无法可靠拆分成多个功能，应明确退化为一次提交，而不是强行拆分。
- 如果某些修改明显属于 bug 修复，不要使用这个 skill，应改用 `fix-gcm`。
